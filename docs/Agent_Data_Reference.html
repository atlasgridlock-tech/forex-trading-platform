🏛️ System Architecture

📋 Trade Lifecycle Flow

The trading system follows a 9-stage pipeline before any order reaches the market. When a scan is triggered, Curator first checks data quality and determines which symbols are tradeable. For each tradeable pair, Compass classifies the current market regime (trending, ranging, breakout-ready). Then five specialist agents analyze the symbol in parallel: Atlas Jr. provides technical analysis and trend grades, Architect maps key support/resistance structures, Oracle assesses the macro outlook, Pulse gauges retail sentiment, and Sentinel flags upcoming event risks. This multi-agent analysis feeds into Tactician, which evaluates strategy templates and generates a setup only if its internal score reaches 70 or higher. The setup then passes through three screening gates: Guardian validates risk parameters and sizes the position, Balancer checks portfolio exposure limits, and Executor confirms the MT5 bridge is ready and spreads are acceptable. At Stage 8, the orchestrator calculates a confluence score (0-100) by weighting all agent inputs—this is the final decision gate requiring 75+ to execute. If approved, the order routes to Executor, which enforces six mandatory safety checks (stop loss required, no martingale, no averaging down, no duplicates, rate limits, Guardian re-approval) before writing the command to a JSON file bridge that the MT5 Expert Advisor reads, executes, and responds to with fill confirmation.

📊 Atlas Jr. — Technical Analysis Agent

📊
Atlas Jr. :3012
Multi-timeframe technical analysis and directional bias

Primary Role

Atlas Jr. is the direction decider for trend strategies. Its directional_lean output determines whether the system looks for LONG or SHORT setups in trending markets.

📥 Inputs
  • OHLCV candle data (from Curator)
  • 500 bars per timeframe
  • 7 timeframes: M1, M5, M15, M30, H1, H4, D1
📤 Outputs
  • directional_lean: bullish / bearish / neutral
  • trend_grade: A / B / C / D / F
  • confidence: 0-100
  • mtf_alignment: ALIGNED_BULLISH / etc
  • emas: {ema8, ema21, ema50, ema200}
  • condition: normal / stretched / compression
  • entry_style: recommendation

Indicators Calculated

CategoryIndicatorsPurpose
TrendEMA 8, 21, 50, 200Direction and trend strength
MomentumRSI (14), MACD, StochasticOverbought/oversold
VolatilityATR (14), Bollinger BandsVolatility state
StrengthADX, +DI, -DITrend strength (>25 = trending)

How Direction Is Determined

For each timeframe (M30, H1, H4, D1): │ ├── Count bullish signals: │ • EMA alignment (8 > 21 > 50): +2 │ • Price above all EMAs: +1 │ • RSI > 55: +1 │ • MACD histogram > 0: +1 │ • +DI > -DI: +1 │ ├── Count bearish signals (opposite) │ └── Determine bias per TF: • bullish_signals > bearish + 2 → "bullish" • bearish_signals > bullish + 2 → "bearish" • else → "neutral" Final directional_lean: • Majority of TFs bullish → "bullish" • Majority of TFs bearish → "bearish" • Mixed → "neutral"

🏛️ Architect — Structure Analysis Agent

🏛️
Architect :3014
Market structure, key levels, FVGs, and invalidation

Primary Role

Architect identifies WHERE price is in relation to key structural levels. Used by Range Fade strategies to determine direction (fade resistance/support) and by all strategies for stop placement.

📥 Inputs
  • H1 and H4 candle data
  • Current price
📤 Outputs
  • structural_bias: bullish / bearish / neutral
  • key_zones[]: Array of S/R levels with freshness
  • fvgs[]: Array of Fair Value Gaps
  • recent_swings[]: HH, HL, LH, LL labels
  • invalidation: {bullish, bearish} levels
  • liquidity_sweeps[]: Detected sweeps

Key Zone Detection

For each swing point in last 10: │ ├── Create zone with buffer (10 pips) │ zone = { │ price: swing_price, │ type: "resistance" or "support", │ upper: price + buffer, │ lower: price - buffer │ } │ ├── Count tests (candles touching zone) │ • 0 tests → "fresh" │ • 1 test → "tested_once" │ • 2+ tests → "tested_multiple" │ └── Check if broken • Price closed beyond zone → "broken" (skip in analysis)

FVG Detection

For 3 consecutive candles [C1, C2, C3]: BULLISH FVG (gap up): If C3.low > C1.high: Gap zone: C1.high to C3.low → Price may return to fill BEARISH FVG (gap down): If C3.high < C1.low: Gap zone: C3.high to C1.low → Price may return to fill

Trading Mode Logic

Check upcoming HIGH impact events: If HIGH impact event within 30 minutes: trading_mode = "PAUSE" → Gate 1 FAILS → NO NEW TRADES If HIGH impact event within 2 hours: trading_mode = "CAUTION" → Reduced confluence score → Wider stops recommended Otherwise: trading_mode = "NORMAL" → Trading allowed

Position Sizing Formula

Account Balance: $10,000 Risk Per Trade: 0.25% = $25 Stop Distance: 30 pips Pip Value (EURUSD): $10 per standard lot Position Size = Risk Amount / (Stop Pips × Pip Value) = $25 / (30 × $10) = $25 / $300 = 0.083 lots → Round to 0.08 lots

🎯 Strategy-Specific Direction Logic

The Key Insight

Different strategies need different direction logic. Trend strategies follow Atlas Jr.; range strategies fade the extremes (contrarian).

Strategy TypeDirection SourceLogic
TREND_CONTINUATIONAtlas Jr. directional_leanFollow the trend
PULLBACK_IN_TRENDAtlas Jr. directional_leanFollow trend, wait for pullback
SESSION_OPEN_DRIVEAtlas Jr. directional_leanFollow session momentum
RANGE_FADEArchitect key_zonesSHORT at resistance, LONG at support
FAILED_BREAKOUTArchitect key_zonesFade false breakouts
BREAKOUTAtlas Jr. or StructureFollow the break direction

Range Fade Direction Logic

check_price_at_zone(price, key_zones, atr): │ ├── Find nearest non-broken zone │ ├── Calculate distance in ATR units: │ distance_atr = |price - zone_price| / ATR │ └── If distance_atr ≤ 0.5 (within 0.5 ATR): │ ├── zone_type == "resistance" │ → Direction: BEARISH (fade it) │ → Check RSI: > 65 confirms, < 35 contradicts │ └── zone_type == "support" → Direction: BULLISH (fade it) → Check RSI: < 35 confirms, > 65 contradicts

🔄 Range Fade System

Zone-Based Take Profits

For range trades, targets are the opposite boundary, not ATR multiples:

RANGE FADE - SHORT at Resistance ═══════════════════════════════════════════════════════════════════════ Resistance: 1.0900 ────●───────── ENTRY (fade it) │ Stop just above (1.0910) │ │ Price moves toward support... ▼ Support 1: 1.0850 ───────────── TP1 (opposite boundary) │ ▼ Support 2: 1.0800 ───────────── TP2 (further support or FVG)

Range Fade SL/TP Logic

For SHORT at resistance: Entry: Current price (at resistance zone) Stop: Resistance zone + (ATR × 0.5) ← Tight stop above zone TP1: First support below price TP2: Second support or unfilled FVG For LONG at support: Entry: Current price (at support zone) Stop: Support zone - (ATR × 0.5) ← Tight stop below zone TP1: First resistance above price TP2: Second resistance or unfilled FVG

⚔️ Conflict Resolution

The Problem

What if TREND says LONG but RANGE_FADE says SHORT?

Conflicting signals cannot both be correct. The system must choose one direction.

The Solution: Regime Decides

If conflict detected (both bullish AND bearish qualified): │ ├── regime == "trending" or "high_vol_expansion" │ └── Trend strategies WIN │ └── Range strategies BLOCKED │ ├── regime == "range_bound" or "mean_reverting" │ └── Range strategies WIN │ └── Trend strategies BLOCKED │ └── regime == "breakout_ready" └── Breakout strategies WIN └── Others BLOCKED

📈 Partial Take Profit System

Exit Framework

Default settings: tp1_r: 1.5 TP1 at 1.5R tp1_pct: 0.5 Close 50% at TP1 tp2_r: 2.5 TP2 at 2.5R tp2_pct: 0.3 Close 30% of original Remaining: 20% runs as "runner"

Why No TP in MT5?

MT5 closes ENTIRE position at TP. For partials:

  1. Order sent with tp: 0
  2. Lifecycle monitors price (1 second)
  3. At TP1: PARTIAL_CLOSE 50%
  4. Move SL to breakeven
  5. At TP2: PARTIAL_CLOSE 60% of remaining
